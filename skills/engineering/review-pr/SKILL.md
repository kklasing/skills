# Review PR

End-to-end review of a GitHub pull request. Combines code review, security review, and convention checks. Posts findings back to GitHub.

## Invocation

`/review-pr <number>` — e.g. `/review-pr 123`. Operates on the current repo's PR. If no number is given, ask the user which PR.

## Workflow

### 1. Fetch the PR

Run `scripts/fetch-pr.sh <number>`. It emits metadata, commits, changed files, and the diff in one pass. If `gh` is missing or unauthenticated, surface the error and stop.

Capture the **head SHA** (`headRefOid`) — required for inline comments.

### 2. Check out the PR locally

To read full files (not just the diff) and to run lint/tests against the PR's HEAD, check it out locally:

```bash
gh pr checkout <number>
```

**This mutates the user's working tree** (switches branch). Before running it:

- [ ] Confirm `git status` is clean — abort and ask if there are uncommitted changes or untracked files that could be lost
- [ ] Note the current branch so you can return to it after the review (`git switch -`)
- [ ] **Ask the user for explicit confirmation** before running the checkout

If the user declines, fall back to `gh api repos/:owner/:repo/contents/<path>?ref=<head_sha>` to read individual files without mutating state — but tests and lint won't be runnable in that mode.

### 3. Read project context

With the PR checked out, ground yourself in the project:

- [ ] Read `CONTEXT.md` if present — domain glossary, terminology
- [ ] Read ADRs (typically `docs/adr/`) that touch the changed areas
- [ ] Open changed files at full size, not just the diff — surrounding code is required to spot cross-file regressions

### 4. Validate PR shape

- [ ] **Conventional commits** — every commit subject must match `<type>(<scope>)?: <subject>`. Allowed types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `perf`, `ci`, `build`, `style`, `revert`. Flag offenders by short SHA.
- [ ] **PR title** — must also be conventional-commit-shaped (it becomes the squash commit on most repos).
- [ ] **CI status** — note any failing required checks; do not approve over them.

### 5. Code review

For every changed hunk:

- Correctness and edge cases
- Conformance to ADRs and CONTEXT.md terminology
- SOLID / KISS / DRY only when violations actually hurt
- Dead code, unused imports, leftover debug, stray TODOs
- **Test coverage** — every behavioural change needs a test; call out missing ones explicitly with the file path where the test belongs

### 6. Security review

- Input validation at trust boundaries
- Authn/authz on new endpoints
- Secrets in code, configs, or logs
- Injection (SQL, command, XSS, SSRF, path traversal)
- Insecure deserialization, unsafe `eval`, prototype pollution
- New dependencies — flag unfamiliar packages and version-pin status

If a real secret appears in the diff, **stop immediately**, do not echo it, and alert the user out-of-band.

### 7. Run tests + linter

Detect tooling from `package.json`, `Makefile`, `pyproject.toml`, etc. Run on the PR's HEAD, in this order:

1. Linter / formatter check
2. Type check (if applicable)
3. Tests

Capture pass/fail. Never auto-fix. If tooling is ambiguous or a job will be slow, confirm before running.

### 8. Post the review

**Inline comments** for findings tied to a specific file/line:

```bash
gh api repos/:owner/:repo/pulls/<number>/comments -X POST \
  -f body='...' -f commit_id='<head_sha>' -f path='<file>' -F line=<n> -f side=RIGHT
```

**Top-level review** with summary, verdict, and rollup:

```bash
gh pr review <number> --comment --body-file review.md
```

Use `--request-changes` for blocking findings. Use `--approve` only if every check passes **and** the user has given explicit go-ahead — never auto-approve.

Every comment posted must start with:

```
> *This was generated by AI during PR review.*
```

### 9. Restore the working tree

If you ran `gh pr checkout` in step 2, switch back to the user's original branch once the review is posted:

```bash
git switch -
```

Confirm `git status` is clean before reporting done.

## Review template

```markdown
> *This was generated by AI during PR review.*

## Summary
<2–3 sentences on what this PR does and the recommendation>

## Blocking
- ...

## Non-blocking suggestions
- ...

## Tests
- <missing coverage, if any>

## Security
- <findings, or "no concerns">

## Conventions & CI
- Conventional commits: ✅ / ❌ (list offenders)
- Lint: <pass/fail>
- Type check: <pass/fail/n/a>
- Tests: <pass/fail>
```

## When to stop and ask

- More than ~10 blocking issues — present a summary first; don't dump a wall of inline comments
- PR scope mismatches the description — flag and stop
- A secret appears in the diff — stop immediately
- The PR is a draft — confirm the user wants a review now anyway
